docker 数据管理

似乎就是一个数据共享的功能，好像也没啥其他的东西了。。

Docker 中的数据卷和挂载是管理容器中数据的两种方法，它们可以用于在容器和主机之间共享数据，或将数据共享给其他容器使用。

  1. 数据卷

数据卷是 Docker 中的一种数据持久化方式，可以将主机或其他容器中的目录或文件挂载到容器内部，使容器内的数据能够持久化保存，并且可以被其他容器访问。数据卷有以下特点：

  • 数据卷是独立于容器的实体，容器可以创建、删除和重建，而数据卷可以一直存在。
  • 数据卷可以被多个容器同时访问，这样可以实现数据的共享。
  • 数据卷的内容可以在容器之间传递，这样可以实现数据的迁移和备份。

可以使用 docker volume create 命令在 Docker 中创建数据卷，然后使用 docker run 命令将数据卷挂载到容器中。

  1. 挂载

挂载是将主机上的目录或文件直接挂载到容器中，使容器可以访问主机上的文件系统，实现数据的共享。挂载有以下特点：

  • 挂载是将主机文件系统直接映射到容器中，因此容器可以直接访问主机上的文件系统。
  • 挂载可以实现容器和主机之间的文件共享和同步。

可以使用 docker run 命令的 -v 或 –mount 选项将主机上的目录或文件挂载到容器中，也可以在 Docker Compose 文件中使用 volumes 来指定挂载点。

需要注意的是，数据卷和挂载都可以用于数据的持久化和共享，但是它们的实现方式不同，因此选择使用哪种方式要根据具体情况来决定。例如，如果需要将数据共享给其他容器使用，则应该选择数据卷；如果需要将数据共享给主机或其他程序使用，则应该选择挂载。

Raft协议是一种分布式一致性算法，旨在解决分布式系统中的数据一致性问题。Raft协议将分布式系统中的节点划分为三种角色：领导者、跟随者和候选人。在正常操作中，所有节点都是跟随者。领导者负责处理客户端请求，将日志项添加到日志中，并向其他节点发送心跳消息以维持自己的领导地位。如果领导者宕机，则会发生选举，跟随者和候选人会通过投票选出新的领导者。

Raft协议的主要特点是可理解性、可管理性和可扩展性。相对于其他一致性算法，Raft协议的设计更加简单，易于理解和实现。同时，Raft协议的管理和维护成本也较低，可以轻松地扩展到大规模分布式系统中。

在实现Raft协议时，需要注意以下几点：

  1. 保证节点间的通信可靠性和顺序性，可以使用TCP协议等通信协议。
  2. 需要处理网络分区和网络延迟等问题，以确保节点之间的一致性。
  3. 需要确保日志项的一致性，即所有节点的日志项必须相同。可以使用复制状态机等技术来实现。
  4. 在选举过程中，需要处理节点崩溃、网络故障等异常情况，以确保选举的正确性和可靠性。

总之，Raft协议是一种可靠、高效、易于理解和实现的分布式一致性算法，已被广泛应用于分布式系统中。

关于新增文件的一些理解

  • 我需要使用cp命令将文件从docker里面复制到本机的对应的有搭建docker的相关文件的文件目录，

命令重新构建一个镜像

  1. 假如我修改的是docker-compose.yml里面的数据，例如端口映射，环境变量等等，我不需要重新构建一个镜像，我只需要重起一个容器就可以了，直接使用up命令会自动重起一个docker，听gpt说应该不会影响docker的网络
  2. 如果我想要清理docker的缓存文件，一般会使用到的关键字是prune，使用这个关键字，可以请悬空镜像，可以清理cache缓存之类的

又一些关于docker的总结

  • 当我使用docker容器的时候其实可以把它当做虚拟机来看到，可以启动和挂起，只要不删除容器，那么总会保留上一次操作的东西，例如从主机当中复制文件到docker中去，也就是说，如果我需要调试一个题目，我只需要基础镜像起一个环境，然后把web服务的文件cp放进docker的相应目录就可以了，不需要修改镜像，调试完毕，删除容器，则基础镜像是不会被影响的
  • 使用docker compose down命令会停止和删除docker容器，会很不方便，如果只是想停止，可以使用docker compose down
  • 关于cp到docker里面的文件，是放在docker的可写层的，可写层也是临时存储层，docker是个具有多层镜像结构的容器
  • 基础镜像层（操作系统：Ubuntu alpine）
  • 应用镜像层（应用程序以及其依赖的库）
  • 可写层（临时文件）
